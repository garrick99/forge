# Forge Language Manual

**Version 1.0 -- March 2026**

Forge is a formally-verified systems programming language that compiles to C99 and CUDA C.
Every program Forge emits is correct by construction: proof obligations are discharged by
a Z3 SMT solver before any code is generated. If the proof fails, the compiler refuses to
emit code. There is no flag to suppress this behavior.

This manual is the comprehensive reference for the Forge language, its standard library,
its proof system, and its toolchain.

---

# Part I: Getting Started

---

## Chapter 1: What is Forge?

Forge is a systems programming language designed for one purpose: to produce programs that
are mathematically guaranteed to be correct. It occupies the space between C (fast, unsafe)
and proof assistants like Coq or Lean (safe, impractical for systems work). Forge gives you
C-level performance with machine-checked correctness proofs on every function.

### The core idea

Every Forge function can carry formal specifications:

- **Preconditions** (`requires`): what must be true when the function is called.
- **Postconditions** (`ensures`): what the function guarantees about its return value.
- **Loop invariants** (`invariant`): what remains true across every iteration of a loop.
- **Termination measures** (`decreases`): proof that loops and recursion always finish.

The Forge compiler generates proof obligations from these specifications and sends them to
the Z3 SMT solver. If Z3 can prove every obligation, the compiler emits C99 (or CUDA C).
If any obligation fails, compilation stops with a diagnostic showing exactly which property
could not be verified.

### What Forge proves

Forge's verification covers the categories of bugs that matter most in systems code:

| Category | Example | How Forge prevents it |
|----------|---------|----------------------|
| Buffer overflow | `s[i]` where `i >= s.len` | Every index generates an `OBoundsCheck` obligation |
| Integer overflow | `a + b` wrapping around | Every arithmetic op generates an `ONoOverflow` obligation |
| Division by zero | `a / b` where `b == 0` | Division generates a precondition `b != 0` |
| Use after free | Reading freed `own<T>` | Linear type system: `own<T>` consumed exactly once |
| Double free | Freeing `own<T>` twice | Linear type system: second use is a compile error |
| Memory leak | Forgetting to free `own<T>` | Linear type system: unconsumed `own<T>` is a compile error |
| Timing side channel | Branching on a secret key | `secret<T>` taint type forbids control-flow dependence |
| GPU deadlock | `syncthreads()` in divergent branch | Barrier placement verified statically |
| Non-termination | Infinite loop or recursion | `decreases` clause checked by Z3 |

### What Forge does not prove

Forge is not a full-spectrum dependent type system. It does not verify:

- Functional correctness of external C functions (declared via `extern`).
- Correctness of inline assembly.
- That your specifications are correct (Forge proves code matches specs, not that specs match intent).
- Liveness properties beyond termination (no deadlock-freedom for concurrent code beyond GPU barriers).

### The output

Forge compiles to clean, readable C99 with no runtime overhead. There are no bounds checks
in the generated code, no garbage collector, no runtime library. The proofs exist only at
compile time and are erased before code generation. The generated C compiles cleanly under
`gcc -Wall -Wextra -Werror`.

```forge
fn divide(a: u64, b: u64) -> u64
    requires b != 0
    ensures  result <= a
{
    a / b
}
```

Generated C:

```c
/* Generated by FORGE compiler -- do not edit.
   All proof obligations discharged. This code is correct by construction. */

#include <stdint.h>
#include <stdbool.h>
#include <stdlib.h>

uint64_t divide(uint64_t a, uint64_t b) {
  return (a / b);
}
```

No runtime check for `b != 0`. The compiler has already proven it at every call site.

### Three-tier proof system

Not every property can be proved automatically by an SMT solver. Forge provides three
tiers of proof support, from fully automatic to fully manual:

| Tier | Mechanism | When to use it |
|------|-----------|----------------|
| **Tier 1** | Z3 SMT (automatic) | Default. Handles all arithmetic, bounds, overflow, and most quantified properties. |
| **Tier 2** | Guided hints | When Z3 needs help: `invariant`, `decreases`, `assert`, `assume`, `ghost` variables. |
| **Tier 3** | Manual proof terms | When Z3 cannot handle nonlinear or inductive properties: `refl`, `symm`, `trans`, `induction`, `by lemma()`. |

Most programs only need Tier 1 and Tier 2. Tier 3 is available for properties like
"the sum formula n*(n+1)/2" or "this recursive function computes the correct value."

## Chapter 2: Installation

### Prerequisites

Forge requires exactly three dependencies:

1. **OCaml 5.0+** -- the compiler is written in OCaml.
2. **Dune 3.0+** -- the build system for OCaml projects.
3. **Z3 4.12+** -- the SMT solver, called as a subprocess (not linked).

No third-party OCaml libraries are needed. No Menhir (the parser is pre-generated).
No ppx_deriving, no fmt, no opam packages beyond Dune itself.

### Installation on Ubuntu / WSL2

```bash
# Install OCaml and opam (if not already present)
sudo apt install opam
opam init
eval $(opam env)

# Install Dune
opam install dune

# Install Z3
sudo apt install z3
# Verify: z3 --version should show 4.12+

# Clone and build Forge
git clone <forge-repo-url>
cd forge
dune build
```

### Installation on macOS

```bash
brew install opam z3
opam init
eval $(opam env)
opam install dune

cd forge
dune build
```

### Installation on Windows

Use WSL2 (Windows Subsystem for Linux) with Ubuntu. Forge builds natively in WSL2
using the same Ubuntu instructions above. The compiler binary is at:

```
./_build/default/bin/main.exe
```

### Verifying the installation

```bash
# Build the compiler
dune build

# Run the simplest demo
./_build/default/bin/main.exe build demos/01_divide.fg
```

You should see output like:

```
  [check] demos/01_divide.fg
  [SMT]    01_divide.fg:4  precondition of '/'
  [SMT]    01_divide.fg:3  postcondition of 'divide'
  all obligations discharged -> emitting C99
```

This confirms the compiler, parser, type checker, proof engine, Z3 integration, and
code generator are all working.

## Chapter 3: Your First Program

### The minimal program

Every Forge program must have at least one function. The simplest complete program:

```forge
fn main() -> u64 { 0u64 }
```

Compile it:

```bash
./_build/default/bin/main.exe build myfile.fg
```

This produces `myfile.c` containing:

```c
#include <stdint.h>
#include <stdbool.h>
#include <stdlib.h>

uint64_t main(void) {
  return 0;
}
```

### Adding a specification

Let us write a function that adds two numbers with overflow protection:

```forge
fn safe_add(a: u64, b: u64) -> u64
    requires a <= 100u64
    requires b <= 100u64
    ensures  result == a + b
    ensures  result <= 200u64
{
    a + b
}

fn main() -> u64 {
    safe_add(50u64, 30u64)
}
```

The `requires` clauses state what callers must guarantee. The `ensures` clauses state
what the function guarantees in return. The compiler generates four proof obligations:

1. At the call site in `main`: `50 <= 100` (first precondition).
2. At the call site in `main`: `30 <= 100` (second precondition).
3. In `safe_add`: the body `a + b` does not overflow (since `a <= 100` and `b <= 100`, the sum is at most 200, which fits in `u64`).
4. In `safe_add`: `result == a + b` and `result <= 200` follow from the body being `a + b` with `a <= 100, b <= 100`.

All four are discharged automatically by Z3.

### Safe division

The classic motivating example. In C, dividing by zero is undefined behavior. In Forge, it
is a compile-time error:

```forge
fn divide(a: u64, b: u64) -> u64
    requires b != 0
    ensures  result <= a
{
    a / b
}

fn main() -> u64 {
    divide(100u64, 3u64)   // Z3 proves: 3 != 0. OK.
}
```

What happens if you try to call `divide` with a potentially-zero divisor?

```forge
fn bad_caller(x: u64) -> u64 {
    divide(10u64, x)   // ERROR: cannot prove x != 0
}
```

The compiler refuses:

```
  FAIL [SMT]  bad_caller.fg:2  precondition of 'divide': b != 0
  proof obligation not discharged -- compilation aborted
```

No C code is emitted. The program does not compile until you prove the divisor is nonzero.

### Proven insertion sort

Here is a complete, verified insertion sort. The compiler proves that the output array
is sorted:

```forge
fn swap_adjacent(s: span<u64>, i: u64)
    requires i + 1u64 < s.len
{
    let tmp: u64 = s[i];
    s[i]        = s[i + 1u64];
    s[i + 1u64] = tmp
}

fn insertion_sort(s: span<u64>)
    ensures forall i: u64, forall j: u64,
              i < j && j < s.len ==> s[i] <= s[j]
{
    let n: u64 = s.len;
    let k: u64 = 0u64;
    while k < n
        invariant k <= n
        invariant forall a: u64, forall b: u64,
                    a < b && b < k ==> s[a] <= s[b]
        decreases n - k
    {
        let j: u64 = k;
        while j > 0u64 && s[j - 1u64] > s[j]
            invariant j <= k
            invariant forall a: u64, forall b: u64,
                        j <= a && a < b && b <= k ==> s[a] <= s[b]
            decreases j
        {
            swap_adjacent(s, j - 1u64);
            j = j - 1u64
        };
        k = k + 1u64
    }
}

fn main() -> u64 { 0u64 }
```

The postcondition `forall i, j: i < j && j < s.len ==> s[i] <= s[j]` is a universal
statement: for every pair of indices in the array, the earlier element is no greater than
the later. Z3 proves this using the loop invariants as inductive lemmas. All 14 proof
obligations are discharged automatically.

## Chapter 4: How Verification Works

### The compilation pipeline

```
source.fg
  |-- Lex (lexer.mll)
  |-- Parse (parser.mly, Menhir LR(1))
  |-- Resolve imports (use std::math, etc.)
  |-- Type check + generate proof obligations (typecheck.ml)
  |-- Discharge obligations (proof_engine.ml)
  |     |-- Tier 1: Z3 SMT (automatic)
  |     |-- Tier 2: guided hints (invariant, decreases, assert, assume)
  |     |-- Tier 3: manual proof terms (induction, by lemma, refl, etc.)
  |-- Erase proofs (ghost variables, proof blocks removed)
  |-- Code generation
  |     |-- C99 backend (codegen_c.ml)
  |     |-- PTX/CUDA backend (codegen_ptx.ml)
  |-- Output: .c / .cu / .ptx
```

### Proof obligation types

The type checker walks the AST and generates proof obligations. Each obligation has a type,
a source location, and a formula to be proved:

| Obligation | Generated when | Formula |
|------------|----------------|---------|
| `OPrecondition` | A function is called | Caller's context implies callee's `requires` clause |
| `OPostcondition` | A function returns | Function body implies the `ensures` clause |
| `OBoundsCheck` | A span or array is indexed | `index < length` |
| `ONoOverflow` | Arithmetic is performed | Result fits in the type's range |
| `OTermination` | A recursive call or loop iteration | The `decreases` measure strictly decreased |
| `OLinear` | An `own<T>` value exists | The value is consumed exactly once |
| `OInvariant` | A loop body executes or a struct is constructed | The `invariant` clause or field predicate holds |

### Z3 integration

Forge translates proof obligations into SMT-LIB 2 queries and sends them to Z3 as a
subprocess. Z3 is not linked as a library; it runs as a separate process. This keeps the
compiler dependency-free and makes it easy to upgrade Z3 independently.

Forge uses two Z3 modes:

- **Integer mode** (default): Variables are modeled as unbounded integers (`Int` sort).
  This is cheaper for most arithmetic properties and handles quantifiers well.

- **Bitvector mode**: Activated when any bitwise operator (`&`, `|`, `^`, `~`, `<<`, `>>`)
  appears in the function. Variables become `(_ BitVec 64)` sorts, giving exact
  modular wrap-around semantics. This is essential for bit-manipulation code.

### Cross-function proof composition

When you call a function that has postconditions, those postconditions become known facts
in the caller. This is how proofs compose across function boundaries:

```forge
fn clamp(v: u64, lo: u64, hi: u64) -> u64
    requires lo <= hi
    ensures  result >= lo
    ensures  result <= hi
{
    if v < lo { lo } else if v > hi { hi } else { v }
}

fn use_clamped(x: u64) -> u64
    ensures result <= 100u64
{
    let c: u64 = clamp(x, 0u64, 100u64);
    // After this call, Z3 knows: c >= 0 AND c <= 100
    // because clamp's postconditions are injected as facts.
    c
}
```

The postcondition `result <= 100u64` in `use_clamped` is proved by Z3 using the
injected fact `c <= 100` from `clamp`'s `ensures` clause.

This injection happens in four patterns:

1. `let x = f(args)` -- postconditions substituted with `x` for `result`.
2. `x = f(args)` -- postconditions substituted with the assigned variable.
3. `s[i] = f(args)` -- postconditions substituted with the array write value.
4. `f(args)` -- void call; postconditions added without result substitution.

### Debugging failed proofs

When an obligation fails, Forge reports the exact location and the property:

```
  FAIL [SMT]  myfile.fg:12  postcondition of 'foo': result < n
```

To get the raw SMT-LIB query for debugging:

```bash
FORGE_DUMP_SMT=1 ./_build/default/bin/main.exe build myfile.fg
# Dumps failing queries to /tmp/forge_smt*.smt2
```

You can feed these files directly to Z3 to experiment:

```bash
z3 /tmp/forge_smt_001.smt2
```

## Chapter 5: Types

### Primitive types

| Type | Width | C equivalent | Description |
|------|-------|--------------|-------------|
| `u8` | 8-bit | `uint8_t` | Unsigned byte |
| `u32` | 32-bit | `uint32_t` | Unsigned 32-bit integer |
| `u64` | 64-bit | `uint64_t` | Unsigned 64-bit integer (default) |
| `i32` | 32-bit | `int32_t` | Signed 32-bit integer |
| `i64` | 64-bit | `int64_t` | Signed 64-bit integer |
| `f32` | 32-bit | `float` | IEEE 754 single-precision float |
| `f64` | 64-bit | `double` | IEEE 754 double-precision float |
| `bool` | 1-bit | `bool` | Boolean (`true` / `false`) |
| `usize` | platform | `size_t` | Platform-width unsigned (array length / index) |

Integer literals can have type suffixes to fix their type:

```forge
42u64     // u64
42u32     // u32
42u8      // u8
42i32     // i32
42i64     // i64
0u64      // zero (u64)
3.14      // f64 (default float)
3.14f32   // f32
```

Without a suffix, integer literals default to `u64` and float literals default to `f64`.

### Struct types

Structs are product types with named fields:

```forge
struct Point {
    x: u64,
    y: u64,
}

// Construction requires the `struct` keyword
let p = struct Point { x: 3u64, y: 7u64 };

// Field access with dot notation
let v = p.x;
```

Structs can have invariants via refinement-typed fields:

```forge
struct Positive {
    value: [n: u64 | n > 0],
}

// Construction generates a proof obligation: the provided value > 0
let p = struct Positive { value: 5u64 };  // OK: Z3 proves 5 > 0
```

Structs compile to C structs with the same layout.

### Enum types

Enums are sum types (tagged unions):

```forge
enum Color { Red, Green, Blue }

enum Shape {
    Circle(f64),         // variant with payload
    Rect(f64, f64),      // variant with two payloads
}

enum Option<T> {
    Some(T),
    None,
}

enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

Enums compile to C tagged unions. Pattern matching on enums must be exhaustive -- the
compiler rejects any match expression that does not cover all variants.

### Span types

A `span<T>` is a fat pointer consisting of a pointer and a length:

```forge
fn sum(s: span<u64>, n: u64) -> u64
    requires n <= s.len
{
    // s.len is the proven length of the span
    // s[i] generates a bounds check: i < s.len
    let mut acc: u64 = 0u64;
    let mut i: u64 = 0u64;
    while i < n
        invariant i <= n
        decreases n - i
    {
        acc = acc + s[i];
        i = i + 1u64;
    };
    acc
}
```

The `.len` field is available on every span and carries into the proof context. Every
indexing operation `s[i]` generates a proof obligation `i < s.len`.

Sub-span slicing:

```forge
let sub = s[lo..hi];
// Generates 3 obligations:
//   lo >= 0
//   lo <= hi
//   hi <= s.len
// Injects fact: sub.len == hi - lo
```

### Fixed-size arrays

```forge
let arr: [u64; 4] = [0u64; 4];            // repeat syntax: all elements are 0
let arr2: [u64; 3] = [1u64, 2u64, 3u64];  // literal syntax

// Indexing is bounds-checked by Z3 (statically, not at runtime)
let v = arr[2u64];   // Z3 proves: 2 < 4
```

### Tuple types

Tuples are anonymous product types:

```forge
fn divmod(a: u64, b: u64) -> (u64, u64)
    requires b != 0
    ensures result.0 * b + result.1 == a
    ensures result.1 < b
{
    (a / b, a % b)
}

// Destructuring
let (q, r): (u64, u64) = divmod(100u64, 7u64);
// After this: q == 14, r == 2
// And Z3 knows: q * 7 + r == 100 (from the postcondition)
```

Tuples compile to anonymous C structs with fields `_0`, `_1`, etc.

### Refinement types

Refinement types embed a predicate directly into a type:

```forge
fn positive(x: [n: u64 | n > 0]) -> u64 {
    x - 1    // Safe: Z3 knows x > 0, so x - 1 >= 0 (no underflow)
}

fn caller() -> u64 {
    positive(5u64)    // Z3 checks: 5 > 0. OK.
}
```

Syntax: `[var: type | predicate]`. The variable name is in scope within the predicate.
At every call site, the predicate becomes a proof obligation.

Refinement types can appear anywhere a type can: function parameters, struct fields,
return types, and let bindings.

### Ownership types

Forge has three ownership levels enforced at compile time:

| Type | Kind | Semantics | C equivalent |
|------|------|-----------|--------------|
| `own<T>` | Linear | Must be consumed exactly once | `T*` (malloc/free) |
| `refmut<T>` | Affine | May be used at most once per call | `T*` (mutable pointer) |
| `ref<T>` | Unrestricted | May be used arbitrarily | `const T*` (read-only pointer) |

```forge
// Allocate on the heap (linear)
let lin p: own<u64> = own_alloc(42u64);

// Destructive read: returns value, frees memory
let v: u64 = own_into(p);
// p is consumed -- cannot be used again

// Borrow without consuming
let r: ref<u64> = own_borrow(&p);       // read-only borrow
let rm: refmut<u64> = own_borrow_mut(&mut p);  // mutable borrow
```

### Secret types

The `secret<T>` type marks cryptographically sensitive values:

```forge
let key: secret<u64> = 42u64;

// OK: bitwise operations on secrets
let mask: secret<u64> = key & 0xFFu64;

// COMPILE ERROR: cannot branch on a secret
if key > 0u64 { ... }   // rejected -- timing side channel

// COMPILE ERROR: cannot use secret as array index
s[key]                   // rejected -- cache timing channel

// Explicit declassification at output boundary
let public: u64 = declassify(mask);
```

Secret types compile to `volatile T` in C to prevent compiler optimizations that could
introduce timing variations.

### Generic types

Types can be parameterized:

```forge
enum Option<T> { Some(T), None }
enum Result<T, E> { Ok(T), Err(E) }
struct Pair<T> { first: T, second: T }
```

Generic types are monomorphized: each concrete instantiation becomes a distinct C type.
`Option<u32>` and `Option<u64>` are separate types.

### Const generics

Functions and types can be parameterized by compile-time integer constants:

```forge
fn dot<N: usize>(a: [u64; N], b: [u64; N]) -> u64 {
    let mut acc: u64 = 0u64;
    for i in N {
        acc = acc + a[i] * b[i];
    };
    acc
}

// Call site: explicitly provide N
let v = dot::<4>(arr_a, arr_b);
```

In C output, `N` is passed as a leading `uint64_t` parameter.

### Type aliases

```forge
type Index = u64;
type Matrix = [u64; 256];

fn get(m: Matrix, i: Index) -> u64
    requires i < 256u64
{
    m[i]
}
```

## Chapter 6: Expressions and Statements

### If / else if / else

`if` is an expression in Forge. Both branches must have the same type:

```forge
let x: u64 = if a > b { a } else { b };

// Chains
let clamped: u64 = if v < lo { lo } else if v > hi { hi } else { v };
```

`if` without `else` returns `()` (unit). The compiler uses branch information in proofs:
inside the `then` branch, the condition is known to be true; inside the `else` branch,
it is known to be false.

### Match expressions

Pattern matching on values with exhaustiveness checking:

```forge
// Literal patterns
match x {
    0u64 => "zero",
    1u64 => "one",
    _    => "other",
}

// Enum variant patterns
match shape {
    Shape::Circle(r) => r * r * 3.14159265,
    Shape::Rect(w, h) => w * h,
}

// Pattern guards
match n {
    k if k < 10u64 => small(k),
    k               => large(k),
}

// OR patterns
match c {
    'a' | 'e' | 'i' | 'o' | 'u' => true,
    _                            => false,
}

// Range patterns
match n {
    0u64..=9u64   => "single digit",
    10u64..=99u64 => "two digits",
    _             => "large",
}

// `as` patterns (bind the whole matched value)
match opt {
    Some(v) as s => use_both(s, v),
    None         => 0u64,
}

// Nested patterns
match pair {
    (Some(a), Some(b)) => a + b,
    _                  => 0u64,
}

// if let (desugars to match)
if let Some(v) = maybe_value {
    use_v(v)
} else {
    default()
}
```

All match expressions must be exhaustive. The compiler rejects any match that does not
cover all possible values of the scrutinee type.

### While loops

```forge
while condition
    invariant property_that_holds_every_iteration
    decreases expression_that_strictly_decreases
{
    // body
};   // semicolon required when followed by another expression
```

The `invariant` clause is checked at loop entry and after every iteration. The `decreases`
clause proves termination: the expression must be a non-negative integer that strictly
decreases on each iteration.

Multiple invariants are allowed:

```forge
while i < n
    invariant i <= n
    invariant count <= i
    invariant forall k: u64, k < i ==> s[k] == 0u64
    decreases n - i
{
    s[i] = 0u64;
    count = count + 1u64;
    i = i + 1u64;
};
```

### For loops

Forge has three forms of `for` loop:

```forge
// Index-based: i = 0, 1, ..., n-1
for i in n {
    s[i] = 0u64;
};

// Element-based span iteration
for x in s {
    total = total + x;
};

// Sub-range: i = lo, lo+1, ..., hi-1
for i in lo..hi {
    process(s[i]);
};
```

Index-based `for` loops automatically generate bounds facts: `i < n` is known inside the
body. Element-based iteration desugars to a hidden index variable.

### Loop / break

Value-returning loops:

```forge
let result: u64 = loop {
    // ... compute ...
    if done { break value; };
};
```

### Let bindings

```forge
let x: u64 = 42u64;          // immutable binding
let mut count: u64 = 0u64;   // mutable binding
let lin p: own<u64> = own_alloc(1u64);  // linear binding

// Tuple destructuring
let (q, r): (u64, u64) = divmod(a, b);
```

### Assignments

```forge
x = x + 1u64;     // variable reassignment (requires `let mut`)
s[i] = value;     // span/array element write
```

### Casts

```forge
let b: u8 = (wide_value as u8);
let s: secret<u64> = (public_val as secret<u64>);
```

### Assert

`assert` is a proof stepping-stone. It asks Z3 to prove a property, then adds it to the
downstream proof context:

```forge
fn example(s: span<u64>, n: u64) -> u64
    requires n > 0u64
    requires n <= s.len
{
    assert(n >= 1u64);   // Z3 proves this from n > 0
    // Now n >= 1 is available as a fact for subsequent proofs
    s[n - 1u64]          // bounds check uses the asserted fact
}
```

### Assume

`assume` introduces a fact without proof. It is logged for auditing:

```forge
assume(external_invariant());
// The fact is taken as true; forge audit shows what was assumed
```

Use `forge audit file.c` to dump all assume entries from generated C.

```forge
return expr;   // explicit early return
```

## Chapter 7: Functions

### Basic function syntax

```forge
fn name(param1: Type1, param2: Type2) -> ReturnType
    requires precondition1
    requires precondition2
    ensures  postcondition1
    ensures  postcondition2
    decreases measure
{
    body_expression
}
```

All clauses are optional. A function with no `requires` accepts any arguments. A function
with no `ensures` makes no guarantees. A function with no `decreases` is not recursive
(or its termination is not checked).

### The `result` keyword

In `ensures` clauses, the keyword `result` refers to the function's return value:

```forge
fn max(a: u64, b: u64) -> u64
    ensures result >= a
    ensures result >= b
    ensures result == a || result == b
{
    if a >= b { a } else { b }
}
```

### Multiple requires and ensures

You can have any number of `requires` and `ensures` clauses. They are logically ANDed:

```forge
fn clamp(v: u64, lo: u64, hi: u64) -> u64
    requires lo <= hi
    ensures  result >= lo
    ensures  result <= hi
    ensures  v < lo  ==> result == lo
    ensures  v > hi  ==> result == hi
    ensures  !(v < lo) && !(v > hi) ==> result == v
{
    if v < lo { lo } else if v > hi { hi } else { v }
}
```

### Lemma functions

Lemmas are proof-only functions. They exist to establish mathematical facts that can be
reused in other proofs:

```forge
lemma add_comm(a: u64, b: u64) -> bool
    ensures result == true
{
    proof { auto }
}

// Use a proved lemma in another function
fn caller(a: u64, b: u64) -> bool
    ensures result == true
{
    by add_comm(a, b)
}
```

### Extern functions

Extern functions declare C functions without a body:

```forge
extern fn malloc(size: u64) -> u64 = "<stdlib.h>";
extern fn sqrt(x: f64) -> f64 = "<math.h>";
```

The `= "<header.h>"` convention tells the code generator to emit `#include <header.h>`
and skip redeclaring the function. Extern functions have no proof obligations on their body
(since there is no body), but their declared preconditions are still checked at call sites.

### Impl blocks

Methods can be defined on structs using `impl` blocks:

```forge
struct Point { x: u64, y: u64 }

impl Point {
    fn manhattan(self: ref<Point>) -> u64 {
        (*self).x + (*self).y
    }

    fn scale(self: ref<Point>, factor: u64) -> Point {
        struct Point { x: (*self).x * factor, y: (*self).y * factor }
    }
}

// Call with dot syntax
let p = struct Point { x: 3u64, y: 7u64 };
let m = p.manhattan();
// Desugars to: Point__manhattan(&p)
```

Methods are mangled to `TypeName__method_name` in C output.

### Trait implementations

```forge
trait Hashable {
    fn hash(self: ref<Self>) -> u64;
}

impl Hashable for Point {
    fn hash(self: ref<Point>) -> u64 {
        (*self).x * 31u64 + (*self).y
    }
}

// Call: p.hash() desugars to Point__Hashable__hash(&p)
```

```forge

#[coalesced]    // assert coalesced memory access pattern
#[ind_cpa]      // IND-CPA structural verification
```

## Chapter 8: Proof Specifications

### Preconditions (`requires`)

Preconditions express what must be true when a function is called. At every call site,
the compiler generates a proof obligation that the caller's context implies the
precondition:

```forge
fn safe_div(a: u64, b: u64) -> u64
    requires b != 0
{
    a / b
}

fn caller(x: u64, y: u64) -> u64
    requires y > 5u64
{
    // Z3 must prove: y > 5 implies y != 0. This succeeds.
    safe_div(x, y)
}
```

### Postconditions (`ensures`)

Postconditions express what the function guarantees about its return value and side effects.
The keyword `result` refers to the return value:

```forge
fn abs_diff(a: u64, b: u64) -> u64
    ensures a >= b ==> result == a - b
    ensures b > a  ==> result == b - a
{
    if a >= b { a - b } else { b - a }
}
```

### Conditional postconditions

The `==>` (implies) operator is commonly used in postconditions to express conditional
guarantees:

```forge
fn find(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    ensures result <= n
    ensures result < n ==> s[result] == target
{
    // ...
}
```

This says: the result is at most `n`, and if it is strictly less than `n`, then the
element at that index equals the target. This is the standard binary search contract.

### Universal quantifiers (`forall`)

Postconditions and invariants can contain universal quantifiers:

```forge
fn fill_zeros(s: span<u64>, n: u64)
    requires n <= s.len
    ensures forall k: u64, k < n ==> s[k] == 0u64
{
    let mut i: u64 = 0u64;
    while i < n
        invariant i <= n
        invariant forall k: u64, k < i ==> s[k] == 0u64
        decreases n - i
    {
        s[i] = 0u64;
        i = i + 1u64;
    };
}
```

The `forall` quantifier states a property for all values of the bound variable. Z3 handles
these using E-matching and model-based quantifier instantiation (MBQI).

Best practice for quantified sorted predicates -- use `&&` format for better trigger matching:

```forge
// Good: && format
requires forall i: u64, forall j: u64, i < j && j < n ==> s[i] <= s[j]

// Less effective: nested ==>
requires forall i: u64, i < j ==> j < n ==> s[i] <= s[j]
```

### Existential quantifiers (`exists`)

```forge
fn has_zero(s: span<u64>, n: u64) -> bool
    requires n <= s.len
    ensures result == true ==> exists k: u64, k < n && s[k] == 0u64
{
    // ...
}
```

### Pre-state references (`old()`)

The `old()` expression captures a value as it was at function entry. This is essential for
specifying the behavior of functions with mutable parameters:

```forge
fn vec_push(v: refmut<Vec_u64>, val: u64)
    requires (*v).len < (*v).data.len
    ensures  (*v).len == old((*v).len) + 1u64
{
    (*v).data[(*v).len] = val;
    (*v).len = (*v).len + 1u64;
}
```

`old((*v).len)` evaluates to the value of `(*v).len` at the moment the function was
called, before any mutations. This lets you express "the length increased by one."

### Ghost variables

Ghost variables exist only in proofs. They are erased before code generation and have
zero runtime cost:

```forge
fn verified_sum(s: span<u64>, n: u64) -> u64
    requires n <= s.len
{
    ghost let sum: u64 = 0u64;
    let mut acc: u64 = 0u64;
    let mut i: u64 = 0u64;
    while i < n
        invariant i <= n
        invariant ghost_sum == acc   // ghost variable tracks real accumulator
        decreases n - i
    {
        acc = acc + s[i];
        ghost sum = sum + s[i];
        i = i + 1u64;
    };
    acc
}
```

Ghost variables are useful for tracking quantities that are needed in proofs but do
not affect the computation.

### Assert as a proof stepping-stone

When Z3 struggles with a complex proof, you can break it into steps using `assert`:

```forge
fn example(a: u64, b: u64, c: u64) -> u64
    requires a < 100u64
    requires b < 100u64
    requires c < 100u64
    ensures result < 300u64
{
    let x: u64 = a + b;
    assert(x < 200u64);       // Z3 proves this from a < 100, b < 100
    let y: u64 = x + c;
    assert(y < 300u64);       // Z3 proves this from x < 200, c < 100
    y
}
```

Each `assert` is proved by Z3, then added to the context for subsequent proofs.

### Assume for external facts

When you need a fact that Forge cannot prove (e.g., a hardware guarantee), use `assume`:

```forge
fn hardware_timer() -> u64 {
    assume(timer_value < 1000000u64);
    // timer_value < 1000000 is now in the proof context
    timer_value
}
```

Every `assume` is logged. Use `forge audit file.c` to review all assumptions in
generated code. In a high-assurance system, the goal is to have zero assumes.

## Chapter 9: Loop Invariants and Termination

### Writing loop invariants

A loop invariant is a property that:

1. Holds before the first iteration (loop entry).
2. Is preserved by each iteration (if it holds at the start of the body, it holds after).
3. Combined with the negation of the loop condition, implies the desired post-loop property.

```forge
fn count_zeros(s: span<u64>, n: u64) -> u64
    requires n <= s.len
    ensures result <= n
{
    let mut count: u64 = 0u64;
    let mut i: u64 = 0u64;
    while i < n
        invariant i <= n           // bounds safety
        invariant count <= i       // count never exceeds elements seen
        decreases n - i            // termination
    {
        if s[i] == 0u64 {
            count = count + 1u64;
        };
        i = i + 1u64;
    };
    // After loop: i == n (from loop exit + invariant i <= n)
    // Therefore: count <= i == n, so count <= n. QED.
    count
}
```

### Quantified loop invariants

The most powerful loop invariants contain quantifiers. They express properties about
all elements processed so far:

```forge
fn fill(s: span<u64>, v: u64, n: u64)
    requires n <= s.len
    ensures forall k: u64, k < n ==> s[k] == v
{
    let mut i: u64 = 0u64;
    while i < n
        invariant i <= n
        invariant forall k: u64, k < i ==> s[k] == v
        decreases n - i
    {
        s[i] = v;
        i = i + 1u64;
    };
    // After loop: i == n.
    // Invariant: forall k < i ==> s[k] == v.
    // Therefore: forall k < n ==> s[k] == v. QED.
}
```

### Nested loop invariants

When loops are nested, each loop has its own invariants:

```forge
fn insertion_sort(s: span<u64>)
    ensures forall i: u64, forall j: u64,
              i < j && j < s.len ==> s[i] <= s[j]
{
    let n: u64 = s.len;
    let k: u64 = 0u64;
    while k < n
        invariant k <= n
        invariant forall a: u64, forall b: u64,
                    a < b && b < k ==> s[a] <= s[b]
        decreases n - k
    {
        let j: u64 = k;
        while j > 0u64 && s[j - 1u64] > s[j]
            invariant j <= k
            invariant forall a: u64, forall b: u64,
                        j <= a && a < b && b <= k ==> s[a] <= s[b]
            decreases j
        {
            // swap s[j-1] and s[j]
            let tmp: u64 = s[j - 1u64];
            s[j - 1u64] = s[j];
            s[j] = tmp;
            j = j - 1u64;
        };
        k = k + 1u64;
    };
}
```

The outer invariant says "the prefix `[0..k)` is sorted." The inner invariant says
"the suffix `[j..k]` is sorted." Together they establish that after the inner loop,
`[0..k+1)` is sorted.

### Termination: `decreases` clauses

Every loop and recursive function must have a `decreases` clause to prove termination.
The expression must be a non-negative integer that strictly decreases on each iteration
or recursive call:

```forge
// Simple loop termination
while i < n
    decreases n - i    // n - i decreases by 1 each iteration
{
    i = i + 1u64;
};

// Recursive termination
fn gcd(a: u64, b: u64) -> u64
    requires a > 0u64
    decreases b
{
    if b == 0u64 { a }
    else { gcd(b, a % b) }  // a % b < b, so the measure decreases
}
```

### Lexicographic termination

For functions where no single measure suffices, use a tuple:

```forge
fn ackermann(m: u64, n: u64) -> u64
    decreases (m, n)    // lexicographic: first compare m, then n
{
    if m == 0u64 { n + 1u64 }
    else if n == 0u64 { ackermann(m - 1u64, 1u64) }
    else { ackermann(m - 1u64, ackermann(m, n - 1u64)) }
}
```

### Mutual recursion

Mutually recursive functions are detected by Tarjan SCC analysis. Each strongly connected
component shares a termination measure:

```forge
fn is_even(n: u64) -> bool
    decreases n
{
    if n == 0u64 { true } else { is_odd(n - 1u64) }
}

fn is_odd(n: u64) -> bool
    decreases n
{
    if n == 0u64 { false } else { is_even(n - 1u64) }
}
```

## Chapter 10: Ownership and Borrows

### The ownership model

Forge's ownership system prevents memory safety bugs at compile time. There are three
ownership kinds:

| Kind | Type | Rule | Violation |
|------|------|------|-----------|
| Linear | `own<T>` | Must be consumed exactly once | Compile error: unconsumed or double-used |
| Affine | `refmut<T>` | May be used at most once per call | Compile error: used twice |
| Unrestricted | `ref<T>` | May be used any number of times | (No restriction) |

### `own<T>`: linear heap ownership

`own<T>` represents a heap-allocated value that the current scope is responsible for
freeing. The compiler ensures it is consumed exactly once:

```forge
fn round_trip(n: u64) -> u64 {
    let lin p: own<u64> = own_alloc(n);   // allocate (calls malloc)
    own_into(p)                            // consume: read value + free memory
}

fn allocate_and_discard(n: u64) -> u64 {
    let lin p: own<u64> = own_alloc(n);
    own_free(p);     // consume: free without reading
    0u64
}
```

The `lin` qualifier on the `let` binding marks the variable as linear. The compiler
tracks that `p` is consumed exactly once (by either `own_into` or `own_free`).

Failing to consume an `own<T>`:

```forge
fn leak(n: u64) -> u64 {
    let lin p: own<u64> = own_alloc(n);
    // ERROR: linear variable 'p' is never consumed
    0u64
}
```

Using an `own<T>` twice:

```forge
fn double_free(n: u64) -> u64 {
    let lin p: own<u64> = own_alloc(n);
    let v: u64 = own_into(p);   // p consumed here
    own_free(p);                 // ERROR: p already consumed
    v
}
```

### `own<T>` API

| Function | Signature | Description |
|----------|-----------|-------------|
| `own_alloc(val)` | `T -> own<T>` | Heap-allocate, initialize with `val` |
| `own_into(p)` | `own<T> -> T` | Read value, free memory (consumes `p`) |
| `own_free(p)` | `own<T> -> ()` | Free memory without reading (consumes `p`) |
| `own_borrow(&p)` | `&own<T> -> ref<T>` | Read-only borrow (does not consume `p`) |
| `own_borrow_mut(&mut p)` | `&mut own<T> -> refmut<T>` | Mutable borrow (does not consume `p`) |
| `own_get(p)` | `own<T> -> (T, own<T>)` | Peek: read value, return pointer back |

### `ref<T>` and `refmut<T>`: borrows

Borrows let you access a value without consuming its ownership:

```forge
fn read_value(x: ref<u64>) -> u64 {
    *x    // dereference: read the pointed-to value
}

fn increment(x: refmut<u64>) {
    *x = *x + 1u64;   // write through mutable borrow
}
```

In C output:
- `ref<T>` becomes `const T*`
- `refmut<T>` becomes `T*`

### Ownership in practice

A typical pattern: allocate, borrow for operations, then free:

```forge
fn process() -> u64 {
    let lin node: own<u64> = own_alloc(100u64);

    // Borrow for reading (does not consume)
    let r: ref<u64> = own_borrow(&node);
    let val: u64 = *r;

    // Now consume the linear value
    own_free(node);

    val
}
```

```forge
fn hash_all<T: Hashable>(items: span<T>, n: u64) -> u64
```

Saturating operations clamp at the type boundary instead of overflowing:


## Chapter 13: std::sort and std::search

Import: `use std::sort;` and `use std::search;`

These modules are the crown jewels of Forge's standard library. They provide sorting
and searching algorithms with mathematically proven correctness.

### Insertion sort (proven sorted)

```forge
fn insertion_sort(s: span<u64>, n: u64)
    requires n <= s.len
    ensures forall i: u64, forall j: u64, i < j && j < n ==> s[i] <= s[j]
```

The postcondition is a universal statement: for every pair of indices `(i, j)` with
`i < j < n`, the element at `i` is at most the element at `j`. This is the mathematical
definition of a sorted array. Z3 proves it using the nested loop invariants.

### Selection sort (proven sorted)

```forge
fn selection_sort(s: span<u64>, n: u64)
    requires n <= s.len
    ensures forall i: u64, forall j: u64, i < j && j < n ==> s[i] <= s[j]
```

Same postcondition, different algorithm. Both are correct by construction.

### Small sorts

```forge
fn sort2(s: span<u64>)
    requires 2u64 <= s.len
    ensures s[0u64] <= s[1u64]

fn sort3(s: span<u64>)
    requires 3u64 <= s.len
    ensures s[0u64] <= s[1u64]
    ensures s[1u64] <= s[2u64]
```

### is_sorted

```forge
fn is_sorted(s: span<u64>, n: u64) -> bool
    requires n <= s.len
    requires n >= 1u64
```

### Binary search (full soundness)

```forge
fn binary_search(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    requires forall i: u64, forall j: u64, i < j && j < n ==> s[i] <= s[j]
    ensures result <= n
    ensures result < n ==> s[result] == target
```

The precondition requires the input to be sorted. The postcondition guarantees: if
the result is a valid index, then that element equals the target. This composes
naturally with `insertion_sort`:

```forge
fn sort_then_find(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    ensures result <= n
    ensures result < n ==> s[result] == target
{
    insertion_sort(s, n);
    // After sort: forall i < j < n ==> s[i] <= s[j]
    // This satisfies binary_search's precondition.
    binary_search(s, n, target)
}
```

| Function | Signature | Postcondition |
|----------|-----------|---------------|

### Using secret types safely

```forge
// Constant-time conditional select (no branching)
fn ct_select(flag: secret<u64>, a: secret<u64>, b: secret<u64>) -> secret<u64> {
    let mask: secret<u64> = 0u64 - flag;
    (mask & a) | (~mask & b)
}

// Constant-time equality (no branching)
fn ct_eq(a: secret<u64>, b: secret<u64>) -> secret<u64> {
    let diff: secret<u64> = a ^ b;
    let neg_diff: secret<u64> = 0u64 - diff;
    let x: secret<u64> = diff | neg_diff;
    1u64 - (x >> 63u64)
}

// Declassify at the output boundary
fn verify_key(key: secret<u64>, candidate: u64) -> bool {
    let eq: secret<u64> = ct_eq(key, candidate as secret<u64>);
    declassify(eq) != 0u64
}
```

### IND-CPA verification

The `#[ind_cpa]` attribute enables structural verification for IND-CPA security.
Inside an `#[ind_cpa]` function, directly declassifying a key is a compile error:

```forge
#[ind_cpa]

    // declassify(key)   // ERROR: key leakage caught by #[ind_cpa]
    let ct: secret<u64> = key ^ (plaintext as secret<u64>);
    declassify(ct)
}
```

## Chapter 19: GPU Backend

Forge compiles `#[kernel]` functions to CUDA C (`__global__` kernels) and optionally
to PTX assembly (SM_89).

### Writing a kernel

```forge
#[kernel]
#[coalesced]
fn vec_add(src_a: span<f32>, src_b: span<f32>, dst: span<f32>)
    requires src_a.len == dst.len
    requires src_b.len == dst.len
    requires dst.len > 0
{
    let i: u64 = blockIdx_x * blockDim_x + threadIdx_x;
    if i < dst.len {
        dst[i] = src_a[i] + src_b[i];
    }
}
```

### GPU built-in variables

| Forge name | CUDA equivalent | Description |
|------------|-----------------|-------------|
| `threadIdx_x` | `threadIdx.x` | Thread index within block (x) |
| `threadIdx_y` | `threadIdx.y` | Thread index within block (y) |
| `blockIdx_x` | `blockIdx.x` | Block index within grid (x) |
| `blockIdx_y` | `blockIdx.y` | Block index within grid (y) |
| `blockDim_x` | `blockDim.x` | Threads per block (x) |
| `gridDim_x` | `gridDim.x` | Blocks per grid (x) |

### Shared memory

```forge
#[kernel]
fn reduce_sum(src: span<u64>, dst: span<u64>,
              num_blocks: u64, block_size: u64)
    requires src.len == num_blocks * block_size
    requires dst.len >= num_blocks
    requires block_size == 256
    requires block_size == blockDim_x
{
    let smem: shared<u64>[256] = 0;

    let tid: u64 = threadIdx_x;
    let bid: u64 = blockIdx_x;
    let gid: u64 = bid * block_size + tid;

    // Phase 1: load into shared memory
    if gid < src.len {
        smem[tid] = src[gid];
    };
    syncthreads();

    // Phase 2: tree reduction
    let stride: u64 = block_size / 2;
    while stride > 0
        invariant stride >= 0
        decreases stride
    {
        if tid < stride {
            smem[tid] = smem[tid] + smem[tid + stride];
        };
        syncthreads();
        stride = stride / 2;
    };

    // Phase 3: write result
    if tid == 0 {
        if bid < dst.len {
            dst[bid] = smem[0];
        };
    };
}
```

### Barrier verification

Forge verifies that `syncthreads()` is never called inside a branch that depends on
a per-thread value. This prevents GPU deadlocks:

```forge
#[kernel]
fn bad_kernel(data: span<u64>) {
    let tid: u64 = threadIdx_x;
    if tid < 128 {
        syncthreads();   // COMPILE ERROR: divergent barrier
    }
}
```

The compiler tracks which values are "varying" (different per thread) vs. "uniform"
(same across all threads in a block). `threadIdx_x` is varying; `blockIdx_x` and
`blockDim_x` are uniform.

### Coalescing analysis

The `#[coalesced]` attribute asserts that memory accesses follow a coalesced pattern
(consecutive threads access consecutive memory locations). The compiler checks this
statically by verifying that array indices depend linearly on `threadIdx_x`.

### PTX output

Forge can emit PTX assembly directly (SM_89 target). The PTX backend handles register
allocation, predicated execution, and shared memory declarations. Output files have
the `.ptx` extension.

### Output files

| Input | Output |
|-------|--------|
| Regular `.fg` | `.c` (C99) |
| `.fg` with `#[kernel]` | `.cu` (CUDA C) |
| `.fg` with PTX target | `.ptx` (PTX assembly) |

## Chapter 21: Cross-Function Proof Composition

One of Forge's most powerful features is automatic proof composition across function
boundaries. When you call a function with postconditions, those postconditions become
known facts in the calling context.

### The basic pattern

```forge
fn safe_sqrt(n: u64) -> u64
    ensures result * result <= n
{
    // ... implementation ...
}

fn use_sqrt(x: u64) -> u64
    requires x <= 100u64
    ensures result <= 100u64
{
    let r: u64 = safe_sqrt(x);
    // Z3 now knows: r * r <= x (from safe_sqrt's postcondition)
    // And: x <= 100 (from this function's precondition)
    // Therefore: r * r <= 100, so r <= 10 <= 100
    r
}
```

### Composition with sort + search

The canonical example of proof composition is sorting an array and then searching it:

```forge
use std::sort;
use std::search;

fn sort_then_find(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    ensures result <= n
    ensures result < n ==> s[result] == target
{
    insertion_sort(s, n);
    // After sort, Z3 knows:
    //   forall i j, i < j && j < n ==> s[i] <= s[j]
    // This is exactly binary_search's precondition.
    binary_search(s, n, target)
    // binary_search's postcondition gives us:
    //   result <= n
    //   result < n ==> s[result] == target
    // Which is exactly our postcondition. QED.
}
```

No manual proof needed. The postconditions compose automatically.

### The four injection patterns

Postconditions are injected in four syntactic patterns:

**Pattern 1: `let x = f(args)`**

```forge
let x = clamp(v, 0u64, 100u64);
// Injected: x >= 0 AND x <= 100
```

**Pattern 2: `x = f(args)`**

```forge
x = clamp(v, 0u64, 100u64);
// Injected: x >= 0 AND x <= 100
```

**Pattern 3: `s[i] = f(args)`**

```forge
s[i] = clamp(v, 0u64, 100u64);
// Injected: s[i] >= 0 AND s[i] <= 100
```

**Pattern 4: `f(args)` (void call)**

```forge
insertion_sort(s, n);
// Injected: forall i j, i < j && j < n ==> s[i] <= s[j]
```

### Limitations

Postcondition injection does NOT work when the call is nested inside an `if` branch:

```forge
// BAD: postcondition not injected (call inside if body)
if cond {
    x = f(args);   // postcondition lost
}
```

Workaround: hoist the call before the branch:

```forge
// GOOD: postcondition injected
let result = f(args);    // postcondition available here
if cond {
    x = result;
}
```

## Chapter 22: Recursive Verification

### Simple recursion with decreases

```forge
fn factorial(n: u64) -> u64
    decreases n
{
    if n == 0u64 { 1u64 }
    else { n * factorial(n - 1u64) }
}
```

The `decreases n` clause tells the prover that `n` strictly decreases on each recursive
call. Since `n` is a non-negative integer, the recursion must terminate.

### Induction hypotheses

For recursive functions with `decreases` clauses, Forge automatically injects the
function's own postconditions as induction hypotheses at recursive call sites. This is
sound by strong induction on the decreasing measure:

```forge
fn sum_to(n: u64) -> u64
    ensures result >= n
    decreases n
{
    if n == 0u64 { 0u64 }
    else {
        let sub: u64 = sum_to(n - 1u64);
        // Induction hypothesis injected: sub >= n - 1
        // Therefore: sub + n >= n - 1 + n >= n. QED.
        sub + n
    }
}
```

### Manual induction proof terms (Tier 3)

When Z3 cannot handle the induction automatically, use explicit proof terms:

```forge
fn prove_nonneg(n: u64) -> bool
    ensures result == true
{
    proof {
        lemma nonneg(n: u64): n >= 0 {
            induction n { auto, auto }
        }
    };
    true
}

fn prove_consec_even(n: u64) -> bool
    ensures result == true
{
    proof {
        lemma consec_even(n: u64): (n * (n + 1)) % 2 == 0 {
            induction n { auto, auto }
        }
    };
    true
}
```

The `induction n { base_proof, step_proof }` form:
- **Base case**: proves `P[n := 0]` using `base_proof`.
- **Step case**: proves `P[n := n+1]` using `step_proof`, with the induction hypothesis `P[n := n]` in context.

### Chaining lemmas

Proved lemmas can be used in other proofs:

```forge
proof {
    lemma step_pos(n: u64): n + 1 > 0 {
        induction n { auto, auto }
    }
    lemma succ_ge_one(n: u64): n + 1 >= 1 {
        by step_pos()
    }
}
```

### Mutual recursion

Mutually recursive functions are detected automatically. Each group shares a termination
measure:

```forge
fn is_even(n: u64) -> bool
    decreases n
{
    if n == 0u64 { true } else { is_odd(n - 1u64) }
}

fn is_odd(n: u64) -> bool
    decreases n
{
    if n == 0u64 { false } else { is_even(n - 1u64) }
}
```

## Chapter 23: Common Proof Patterns and Troubleshooting

### Pattern 1: Quantified sorted invariant

When proving sortedness, use the `&&` conjunction format:

```forge
// Good: Z3 finds triggers easily
invariant forall i: u64, forall j: u64,
    i < j && j < k ==> s[i] <= s[j]

// Bad: nested implications confuse MBQI
invariant forall i: u64, forall j: u64,
    i < j ==> j < k ==> s[i] <= s[j]
```

### Pattern 2: Binary search frontier invariants

Guard with a sentinel check to prevent violation in the "found" branch:

```forge
invariant ans == n ==> forall k: u64, k < lo ==> s[k] < target
invariant ans == n ==> forall k: u64, k >= hi && k < n ==> s[k] > target
```

### Pattern 3: Bitwise bounds

When using bitwise operations, Z3 switches to bitvector mode. Help Z3 with explicit
bounds assertions:

```forge
let masked: u64 = x & 0xFFu64;
assert(masked <= 255u64);   // Z3 in bitvector mode proves this
```

### Pattern 4: Nonlinear arithmetic

Z3 struggles with products. Avoid expressions like `state * nalpha + input[i]` in
proof contexts. Instead:

```forge
// Compute the index and check bounds explicitly
let idx: u64 = state * nalpha + input[i];
if idx < nstates * nalpha {
    table[idx]
}
```

### Pattern 5: Hoisting calls from branches

Postcondition injection does not work inside if-branches. Hoist calls:

```forge
// Bad: postcondition not injected
if cond { x = f(a); } else { x = g(b); };

// Good: postconditions injected for both calls
let ra = f(a);    // postcondition of f available
let rb = g(b);    // postcondition of g available
if cond { x = ra; } else { x = rb; };
```

### Pattern 6: Avoiding boolean flag loops

Boolean flag loops break termination proofs:

```forge
// Bad: done flag does not decrease
let mut done: bool = false;
while i < n && !done
    decreases n - i    // broken: when done=true, i doesn't change
{
    if found { done = true; }
    else { i = i + 1u64; };
};

// Good: use i = n to exit
while i < n
    decreases n - i
{
    if found { i = n; }     // forces loop exit, measure drops to 0
    else { i = i + 1u64; };
};
```

### Pattern 7: ITE-valued writes for conditional swaps

The compiler extracts array writes from direct assignments but not from nested if-blocks:

```forge
// Good: ITE-valued writes (extracted correctly)
let v0: u64 = s[0u64];
let v1: u64 = s[1u64];
s[0u64] = if v0 <= v1 { v0 } else { v1 };
s[1u64] = if v0 <= v1 { v1 } else { v0 };

// Bad: nested if swap (not extracted by the compiler)
if s[0u64] > s[1u64] {
    let tmp = s[0u64];
    s[0u64] = s[1u64];
    s[1u64] = tmp;
}
```

### Pattern 8: Early exit without overflow

Do not use `i = n + 1` to break a `while i < n` loop:

```forge
// Bad: violates invariant i <= n
i = n + 1u64;

// Good: set i = n to exit cleanly
i = n;
```

### Debugging tips

1. **Add assert statements** to break complex proofs into smaller steps.
2. **Check postcondition injection** -- ensure calls are at the top level, not inside branches.
3. **Use `FORGE_DUMP_SMT=1`** to inspect the actual SMT queries Z3 is receiving.
4. **Simplify invariants** -- start with the minimal invariant and add clauses one at a time.
5. **Check bitvector mode** -- if your function uses `&`, `|`, `^`, etc., all arithmetic is in bitvector mode. Add explicit bounds assertions.

## Chapter 24: Building Real Applications

This chapter walks through six complete, verified applications that demonstrate Forge's
capabilities for real-world systems programming.

### Application 1: Verified binary search

The standard binary search, with full proof of correctness:

```forge
fn binary_search(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    requires forall i: u64, forall j: u64, i < j && j < n ==> s[i] <= s[j]
    ensures result <= n
    ensures result < n ==> s[result] == target
{
    let mut lo: u64 = 0u64;
    let mut hi: u64 = n;
    let mut ans: u64 = n;
    while lo < hi
        invariant lo <= hi
        invariant hi <= n
        invariant ans <= n
        invariant ans < n ==> s[ans] == target
        invariant ans == n ==> forall k: u64, k < lo ==> s[k] < target
        invariant ans == n ==> forall k: u64, k >= hi && k < n ==> s[k] > target
        decreases hi - lo
    {
        let mid: u64 = lo + (hi - lo) / 2u64;
        if s[mid] == target {
            ans = mid;
            lo = hi;
        } else if s[mid] < target {
            lo = mid + 1u64;
        } else {
            hi = mid;
        };
    };
    ans
}
```

What Forge proves:
- `mid` is always in bounds (`mid < n`).
- `hi - lo` strictly decreases (termination).
- If `result < n`, then `s[result] == target` (correctness).
- `result <= n` always (safety).

### Application 2: Verified GCD

Euclidean GCD with termination proof:

```forge
fn gcd(a: u64, b: u64) -> u64
    requires a > 0u64
    decreases b
{
    if b == 0u64 { a }
    else { gcd(b, a % b) }
}
```

Forge proves:
- At the recursive call, `b > 0` (from the else branch), so `a % b` is valid.
- The recursive call has `a % b` as the new `b`, and `a % b < b` (Z3 knows this
  property of modular arithmetic), so the measure decreases.

### Application 3: Constant-time modular exponentiation

A cryptographic modular exponentiation that never branches on the secret exponent:

```forge
fn ct_modpow(base: secret<u64>, exp: secret<u64>,
             modulus: u64) -> secret<u64>
    requires modulus > 1u64
{
    let result: secret<u64> = 1u64;
    let b: secret<u64> = base;
    let e: secret<u64> = exp;
    let mut i: u64 = 0u64;
    while i < 64u64
        invariant i <= 64u64
        decreases 64u64 - i
    {
        // Constant-time: always multiply, select based on bit
        let bit: secret<u64> = (e >> i) & 1u64;
        let candidate: secret<u64> = (result * b) % modulus;
        result = ct_select(bit, candidate, result);
        b = (b * b) % modulus;
        i = i + 1u64;
    };
    result
}
```

The `secret<T>` type ensures no branching on `bit`, `e`, or any derived value.
`ct_select` uses bitwise masking instead of an if-statement.

### Application 4: GPU parallel reduction

A complete GPU tree reduction kernel (shown in detail in Chapter 19). Forge verifies:
- Shared memory accesses are in bounds.
- `syncthreads()` is never called in a divergent branch.
- The final write to `dst[bid]` is in bounds.

### Application 5: Verified sorting pipeline

Sort an array, then binary search, with the entire proof chain verified:

```forge
use std::sort;
use std::search;

fn sort_and_find(s: span<u64>, n: u64, target: u64) -> u64
    requires n <= s.len
    ensures result <= n
    ensures result < n ==> s[result] == target
{
    insertion_sort(s, n);
    binary_search(s, n, target)
}
```

Zero proof annotations needed in the caller. The postcondition of `insertion_sort`
(sorted output) satisfies the precondition of `binary_search` (sorted input)
automatically.

### Application 6: LZ77 compression with proven memory safety

The `std::compress` module implements LZ77 compression where every buffer access is
verified. Even on malformed input, no buffer overflow can occur:

```forge
use std::compress;

fn safe_compress(input: span<u64>, in_len: u64,
                 output: span<u64>, out_cap: u64) -> u64
    requires in_len <= input.len
    requires out_cap <= output.len
{
    lz_compress(input, in_len, output, out_cap)
}

fn safe_decompress(tokens: span<u64>, tok_len: u64,

    requires tok_len <= tokens.len
    requires out_cap <= output.len
    ensures result <= out_cap
{
    lz_decompress(tokens, tok_len, output, out_cap)
}
```

The `ensures result <= out_cap` on decompression guarantees that the number of output
elements never exceeds the buffer capacity, regardless of what the token stream contains.

## Chapter 25: The Verified Packet Filter

This chapter demonstrates how Forge can be used to write a verified network packet parser
that eliminates buffer overflow vulnerabilities by construction.

### The problem

Network packet parsers are among the most security-critical pieces of systems software.
A single buffer overflow in a packet parser can give an attacker remote code execution.
CVE databases are full of such vulnerabilities (CWE-120, CWE-125, CWE-131, CWE-805).

### The Forge approach

In Forge, every byte access in the packet buffer is bounds-checked by Z3 at compile time.
The generated C code has zero runtime bounds checks -- the proofs guarantee that every
access is within the buffer.

```forge
fn parse_ipv4(pkt: span<u8>, pkt_len: u64) -> (u64, u64, u64)
    requires pkt_len <= pkt.len
    requires pkt_len >= 20u64    // minimum IPv4 header
{
    // Version and IHL (byte 0)
    let version: u8 = (pkt[0u8] >> 4u8) & 0x0Fu8;
    let ihl: u8 = pkt[0u8] & 0x0Fu8;

    // Total length (bytes 2-3, big-endian)
    let total_len: u64 = (pkt[2u8] as u64) * 256u64 + (pkt[3u8] as u64);

    // Protocol (byte 9)
    let protocol: u64 = pkt[9u8] as u64;

    // Source IP (bytes 12-15)
    let src_ip: u64 = (pkt[12u8] as u64) * 16777216u64
                    + (pkt[13u8] as u64) * 65536u64
                    + (pkt[14u8] as u64) * 256u64
                    + (pkt[15u8] as u64);

    // Destination IP (bytes 16-19)
    let dst_ip: u64 = (pkt[16u8] as u64) * 16777216u64
                    + (pkt[17u8] as u64) * 65536u64
                    + (pkt[18u8] as u64) * 256u64
                    + (pkt[19u8] as u64);

    (total_len, src_ip, dst_ip)
}
```

Every index (0, 2, 3, 9, 12-19) is statically verified to be less than `pkt_len` (which
is at least 20). The generated C has no bounds checks:

```c
uint64_t version = (pkt.data[0] >> 4) & 0x0F;
uint64_t ihl = pkt.data[0] & 0x0F;
// ... direct array accesses, no checks ...
```

### Extending the parser

To parse TCP headers after the IP header:

```forge
fn parse_tcp_ports(pkt: span<u8>, pkt_len: u64, ip_hdr_len: u64)
    -> (u64, u64)
    requires pkt_len <= pkt.len
    requires ip_hdr_len >= 20u64
    requires ip_hdr_len + 4u64 <= pkt_len   // need 4 bytes for ports
{
    let base: u64 = ip_hdr_len;
    let src_port: u64 = (pkt[base] as u64) * 256u64
                      + (pkt[base + 1u64] as u64);
    let dst_port: u64 = (pkt[base + 2u64] as u64) * 256u64
                      + (pkt[base + 3u64] as u64);
    (src_port, dst_port)
}
```

The precondition `ip_hdr_len + 4 <= pkt_len` guarantees that `base`, `base+1`, `base+2`,
and `base+3` are all valid indices.

### What this eliminates

By construction, a Forge packet parser cannot have:
- **CWE-120**: Buffer Copy without Checking Size of Input.
- **CWE-125**: Out-of-bounds Read.
- **CWE-131**: Incorrect Calculation of Buffer Size.
- **CWE-805**: Buffer Access with Incorrect Length Value.

The proofs are part of the source code, checked on every compilation, and the generated
C is as fast as hand-written C.

## Appendix B: Proof Obligation Types

### OPrecondition

Generated at every function call site. The caller's proof context must imply the callee's
`requires` clauses.

```forge
fn divide(a: u64, b: u64) -> u64
    requires b != 0
{ a / b }

fn caller(x: u64) -> u64
    requires x > 5u64
{
    divide(10u64, x)
    // Obligation: x > 5 implies x != 0? YES. Discharged.
}
```

### OPostcondition

Generated at every function exit point. The function body must imply the `ensures` clauses.

```forge
fn max(a: u64, b: u64) -> u64
    ensures result >= a
    ensures result >= b
{
    if a >= b { a } else { b }
    // Branch 1: result == a, and a >= b. So result >= a (trivial) and result >= b (from a >= b).
    // Branch 2: result == b, and b > a. So result >= b (trivial) and result >= a (from b > a).
}
```

### OBoundsCheck

Generated for every span or array index access.

```forge
fn get(s: span<u64>, i: u64) -> u64
    requires i < s.len
{
    s[i]
    // Obligation: i < s.len? Given by precondition. Discharged.
}
```

### ONoOverflow

Generated for arithmetic operations that could overflow or underflow.

```forge
fn safe_add(a: u64, b: u64) -> u64
    requires a <= 100u64
    requires b <= 100u64
{
    a + b
    // Obligation: a + b <= UINT64_MAX? Since a + b <= 200, yes. Discharged.
}
```

### OTermination

Generated for recursive calls and loop iterations.

```forge
fn f(n: u64) -> u64
    decreases n
{
    if n == 0u64 { 0u64 }
    else { f(n - 1u64) }
    // Obligation: n - 1 < n? When n > 0, yes. Discharged.
}
```

### OLinear

Generated for `own<T>` bindings. The value must be consumed exactly once.

```forge
fn use_own() -> u64 {
    let lin p: own<u64> = own_alloc(42u64);
    own_into(p)
    // Obligation: p consumed exactly once? Yes (by own_into). Discharged.
}
```

### OInvariant

Generated at loop body entry, loop body exit, and struct construction.

```forge
while i < n
    invariant i <= n
{
    i = i + 1u64;
    // Obligation: after body, i <= n still holds?
    // Pre-body: i < n AND i <= n. After: i' = i + 1. Since i < n, i + 1 <= n. Discharged.
};
```

## Appendix C: Z3 Integration Details

### How Forge talks to Z3

Forge spawns Z3 as a subprocess and communicates via SMT-LIB 2 commands on stdin/stdout.
Each proof obligation becomes an independent `(check-sat)` query.

### Integer mode

The default. Variables become SMT `Int` sort. Arithmetic is unbounded. This mode is fast
and handles quantifiers well via E-matching.

Example SMT-LIB output:

```smt2
(declare-const a Int)
(declare-const b Int)
(assert (>= a 0))
(assert (>= b 0))
(assert (<= a 100))
(assert (<= b 100))
(assert (not (<= (+ a b) 200)))
(check-sat)
; Expected: unsat (meaning the negation is impossible, so the property holds)
```

### Bitvector mode

Activated when bitwise operators appear. Variables become `(_ BitVec 64)`. Arithmetic
wraps around at 2^64.

Example SMT-LIB output:

```smt2
(declare-const x (_ BitVec 64))
(declare-const masked (_ BitVec 64))
(assert (= masked (bvand x #x00000000000000FF)))
(assert (not (bvule masked #x00000000000000FF)))
(check-sat)
; Expected: unsat
```

### Quantifier handling

Z3 uses MBQI (Model-Based Quantifier Instantiation) for quantified formulas. Forge
structures quantified postconditions to produce good triggers:

```forge
// This generates a trigger on s[k]:
ensures forall k: u64, k < n ==> s[k] == 0u64
```

### Timeouts

Z3 queries have a default timeout. If a query times out, the obligation is reported as
failed. Use `assert` statements to break complex proofs into smaller steps that Z3 can
handle within the timeout.

## Appendix F: Compiler Architecture

### Overview

The Forge compiler is written in approximately 9,700 lines of OCaml, organized into
six modules:

| Module | File | Lines | Purpose |
|--------|------|-------|---------|
| AST | `lib/ast/ast.ml` | ~340 | AST node types |
| Lexer | `lib/lexer/lexer.mll` | ~225 | Tokenization (Ocamllex) |
| Parser | `lib/parser/parser.mly` | ~1125 | LR(1) grammar (Menhir) |
| Type Checker | `lib/types/typecheck.ml` | ~3700 | Type checking + obligation generation |
| Proof Engine | `lib/proof/proof_engine.ml` | ~1225 | Z3 bridge + three-tier discharge |
| C Codegen | `lib/codegen/codegen_c.ml` | ~2150 | C99 code generation |
| PTX Codegen | `lib/codegen/codegen_ptx.ml` | ~620 | PTX/CUDA backend |
| CLI | `bin/main.ml` | ~290 | Command-line driver |

### The pipeline in detail

**Step 1: Lexing.** The Ocamllex-generated lexer converts source text into tokens.

**Step 2: Parsing.** The Menhir-generated LR(1) parser converts tokens into an AST.
The parser is pre-generated and committed to the repository; Menhir is not needed at
build time.

**Step 3: Import resolution.** `use std::math;` loads `./std/math.fg` and prepends its
declarations to the current module.

**Step 4: Type checking and obligation generation.** The type checker walks the AST and:
- Infers and checks types for all expressions.
- Generates proof obligations (OPrecondition, OPostcondition, etc.).
- Tracks linear ownership (own<T> consumed exactly once).
- Injects callee postconditions at call sites.
- Builds the proof context (set of known facts) for each program point.

**Step 5: Proof discharge.** The proof engine processes obligations in order:
1. Try Tier 1 (Z3 SMT). If Z3 returns `unsat` (the negation is impossible), the
   obligation is discharged.
2. If the obligation has a guided hint (invariant, decreases), use it to strengthen
   the proof context before querying Z3.
3. If the obligation has a manual proof term, evaluate the term in the CoC subset.
4. If none of the tiers can discharge the obligation, report a failure and stop.

**Step 6: Proof erasure.** All proof-related AST nodes are removed: ghost variables,
proof blocks, assert/assume statements, specification clauses. The remaining AST
contains only computational content.

**Step 7: Code generation.** The erased AST is translated to C99 (or CUDA C, or PTX).
The output is clean, readable, and compiles without warnings under strict compiler flags.

### Key internal mechanisms

**Cross-function postcondition injection (`inject_postconds`).** When the type checker
encounters a call to a function with `ensures` clauses, it substitutes `result` with the
bound variable and adds the resulting facts to the proof context.

**Recursive induction hypothesis injection (`inject_rec`).** For functions with `decreases`
clauses, the type checker finds recursive calls and injects the function's own
postconditions as assumptions (sound by strong induction on the decreasing measure).

**Array mutation tracking (SSA).** When `s[i] = v` executes, the proof engine renames
`s` to `__s_pN` and adds facts: `s[i] == v` (write fact) and
`forall j, j != i ==> s[j] == __s_prev[j]` (frame fact). This gives Z3 a complete
model of array state across mutations.

**Conditional array writes.** For `if` expressions containing array writes, the proof
engine builds ITE (if-then-else) predicates: `s[i] = cond ? v_then : v_else`. This
handles swap patterns and conditional updates.
